### 1. LLM（大语言模型）

- **解释**：LLM 是一个通过海量文本训练出来的概率模型，能够根据输入的文本预测并生成接下来最可能出现的文字。
- **具体例子**：当你问 ChatGPT “法国的首都是哪里？”，它基于训练数据中的模式，预测出 “巴黎” 作为答案。

### 2. Prompt（提示）

- **解释**：Prompt 是你给 LLM 的输入指令或问题，用来引导模型产生你想要的输出。
- **具体例子**：与其只写“写首诗”，更好的 prompt 是“以秋天为主题，写一首五言绝句，风格忧郁”。
- **常见误区**：很多人以为 prompt 越长、越复杂就一定越好。实际上，过于冗长或矛盾的 prompt 可能使模型困惑；有效的 prompt 往往需要简洁、明确、提供必要的上下文。

### 3. Context Window（上下文窗口）

- **解释**：Context Window 是 LLM 一次能“记住”并处理的输入令牌（token）最大长度，超出部分模型就会遗忘。
- **具体例子**：如果某模型的上下文窗口是 4096 个 token，你给它一本 5000 token 的小说让它续写，模型只能看到前 4096 个 token，后面的 904 个 token 相当于从未存在过。
- **常见误区**：有人以为只要对话在窗口内，模型就能完美利用所有信息。实际上，随着长度增加，模型对中间部分信息的召回能力会下降（“中间迷失”效应）。窗口越大不等于理解越深。

### 4. Agent（智能体）

- **解释**：Agent 是一个能自主规划步骤、调用工具（如搜索引擎、计算器、代码执行器）并循环执行直至完成目标的 LLM 驱动系统。
- **具体例子**：你让 Agent “帮我订一张下周去北京的机票，预算 800 元以内”。它会先拆解任务：查询航班、比较价格、判断是否满足预算、可能调整日期，最后给出推荐甚至自动打开订票网站。
- **常见误区**：Agent 不是“一次回答就完事”的简单聊天机器人。常见误区是把任何带 LLM 调用的流程都叫做 Agent —— 真正的 Agent 必须具备**自主决策**和**多轮行动-观察-调整**的循环能力。边界在于：当前 Agent 容易陷入死循环或错误决策，需要人类监督。

### 5. Tool Use（工具调用）

- **解释**：Tool Use 指 LLM 在生成过程中主动请求调用外部函数或 API（如计算器、数据库查询、代码解释器），并将返回结果纳入推理。
- **具体例子**：用户问“45 × 19 等于多少？”，LLM 不直接输出答案，而是返回一个函数调用 `multiply(45, 19)`，系统计算后把结果 855 交给模型，模型再说出答案。
- **常见误区**：有人以为 LLM 会“自动”使用所有可用的工具。实际上，模型必须经过专门训练（如 Function Calling）才能正确识别何时该调用工具、调用哪个工具。不是任意 LLM 都能可靠使用工具；而且工具描述不清或过多，模型容易选错。

### 6. AI Coding（AI 辅助编程）

- **解释**：AI Coding 是指利用 LLM 来生成、补全、解释、调试或重构代码，提升编程效率的开发方式。
- **具体例子**：在 VS Code 中使用 Copilot，键入函数名 `calculate_average`，AI 自动补全出遍历数组求和并除以长度的代码。
- **常见误区**：常见误区是认为 AI 生成的代码一定正确、安全、高效。实际上，AI 可能引入漏洞、性能问题或不符合项目风格。边界在于：AI Coding 是“强化的自动补全+初级代码审查”，最终责任仍由开发者承担，不能完全替代人工理解与测试。

### 7. Workflow（工作流）

- **解释**：Workflow 是将 LLM 的调用过程拆解为固定的、可重复的步骤序列（如先分类、再检索、最后生成），每一步输出作为下一步输入。
- **具体例子**：一个客服机器人 workflow：  
  ① 将用户问题分类（退货/咨询/投诉）；  
  ② 根据类别从知识库检索相关文档；  
  ③ LLM 基于检索结果生成回答；  
  ④ 检查回答是否包含敏感词。每一步由不同模块或 LLM 调用完成。
- **常见误区**：Workflow ≠ Agent。Workflow 是**预定义**的固定路径，没有自主决策；Agent 会动态选择下一步。误区是以为加几个 if‑else 就是 workflow，其实真正的 workflow 往往需要编排工具（如 LangGraph、Prefect）。边界：Workflow 适合确定性高、步骤清晰的任务；复杂开放任务用 Agent 更灵活但更不可控。

### 8. Guardrails（护栏 / 安全围栏）

- **解释**：Guardrails 是一套在 LLM 输入前或输出后执行的检验规则，用于拦截有害、敏感或不符合规范的提示和回复。
- **具体例子**：用户问“如何制造炸弹”，输入护栏检测到关键词并直接拒绝，不交给 LLM；或者 LLM 输出了“我建议你…”后面含危险内容，输出护栏用另一个模型打分，低于阈值就替换为“抱歉，我无法回答这个问题”。
- **常见误区**：很多人以为只要模型本身对齐了（比如 GPT‑4），就不需要额外护栏。实际上模型可能被越狱 prompt 或间接提示绕过；护栏是**防御层**，不能完全依赖模型自律。边界：护栏过严会导致过度拒绝（用户正常问题被误拦），需要在安全与可用性之间权衡。

### 9. Tracing（追踪）

- **解释**：Tracing 记录 LLM 应用在一次请求中的完整执行轨迹，包括输入、中间步骤、工具调用、耗时、token 消耗等，用于调试和性能分析。
- **具体例子**：一个 Agent 去查天气 → 调用位置 API → 调用天气 API → LLM 生成回答。Tracing 工具（如 LangSmith）会显示每一步的输入输出、延迟、失败重试情况，帮你定位为什么 Agent 回答“我找不到那个城市”。
- **常见误区**：仅打印日志不等于 tracing。真正的 tracing 需要保持**调用链的关联性**（一个 request ID 串联所有跨度）。误区是只在出问题后才加 tracing —— 实际上应该在开发初期就集成，否则复杂 workflow 出错了几乎无法调试。边界：tracing 会引入额外开销（存储、性能），生产环境通常采样记录。

### 10. Human-in-the-loop（人机协同 / 人在回路）

- **解释**：Human-in-the-loop 指在 LLM 自动执行流程的关键节点，引入人工审核、确认或修正，以保证结果的正确性和安全性。
- **具体例子**：一个自动发送邮件的 Agent 生成了一封合同修改建议，不直接发出，而是先推送给部门主管，主管点击“批准”或“修改后重试”后才会真正执行。
- **常见误区**：有人把任何需要人写 prompt 都叫做 human-in-the-loop，这是误解。真正的 HITL 是**流程内嵌的强制审批机制**，通常用于高风险动作（付款、发文、删除数据）。边界在于：过度使用 HITL 会失去自动化价值，因此只应在不确定性高或代价高的步骤加入人工，而不是所有步骤。

### 全部 10 个概念汇总（方便后续查阅）

| 概念 | 一句话解释 |
|------|------------|
| LLM | 基于海量文本训练的概率性文字生成模型 |
| Prompt | 引导 LLM 产生特定输出的输入指令 |
| Context Window | LLM 一次能处理的最大 token 数量 |
| Workflow | 固定的、可分步执行的 LLM 处理流水线 |
| Agent | 能自主规划、调用工具、多轮循环完成目标的系统 |
| Tool Use | LLM 主动请求外部 API 或函数来获取信息或执行操作 |
| AI Coding | 用 LLM 辅助生成、补全、调试代码的开发方式 |
| Guardrails | 对输入/输出进行规则检查，拦截不安全内容 |
| Tracing | 记录一次请求中所有步骤的完整链路，用于调试 |
| Human-in-the-loop | 自动化流程中插入人工确认或修正的环节 |

这些概念彼此关联：**LLM** 是内核，**Prompt** 是控制方式，**Context Window** 是其记忆限制；在此基础上，**Tool Use** 让 LLM 能够与外部世界交互，**Agent** 则是把工具调用和规划循环组合成自主系统，而 **AI Coding** 是上述概念在编程领域的具体应用。**Workflow** 和 **Agent** 是两种不同的编排范式；**Guardrails** 与 **Human-in-the-loop** 是安全与可控性的保障机制；**Tracing** 是观察和调试所有上述系统的必备工具。希望这套完整版能帮你建立扎实的共同语言。

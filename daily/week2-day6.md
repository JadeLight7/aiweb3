# Week 2 Day 6 — Agent Token 采购系统：项目设计

> 2026-05-30

## 一、项目定位

### 一句话定义

**让 AI Agent 能自动购买所需的 token/API 服务，你控制预算，系统记录所有支出。**

### 核心价值主张

| 角色 | 痛点 | 本项目解决方案 |
|------|------|--------------|
| **你（Agent 所有者）** | 不知道 Agent 花了多少钱 | 链上记录所有交易，一键查询 |
| **你（Agent 所有者）** | 怕 Agent 乱花钱 | Pact 限制单笔/每日/每月预算 |
| **Agent** | 每次都要人工买 API key | 自动响应 402，自动支付 |
| **Agent** | 不知道哪个 API 性价比高 | 记录每次购买的 token 类型、价格、用途 |

---

## 二、要解决的问题

### 2.1 详细痛点分析

#### 痛点 1：手动购买 API key
- **现状**：每次 Agent 需要调用付费 API，都要人工操作
- **后果**：Agent 无法自主运行，需要人工干预
- **例子**：Agent 需要调用 OpenAI API，你必须手动去购买 API key
- **本项目方案**：Agent 自动响应 402，自动支付，无需人工干预

#### 痛点 2：不知道花了多少钱
- **现状**：支出分散在各个服务商，没有统一记录
- **后果**：月底才发现超支，无法提前预警
- **例子**：Agent 调用了 10 个不同的付费 API，你不知道总共花了多少
- **本项目方案**：链上记录所有交易，支持按时间、按类型、按服务商查询

#### 痛点 3：怕 Agent 乱花钱
- **现状**：没有预算控制，Agent 可能无限调用付费 API
- **后果**：账单爆炸，无法控制成本
- **例子**：Agent 循环调用某个 API，一晚上花了 1000 美元
- **本项目方案**：Pact 限制单笔/每日/每月预算，超预算自动暂停

#### 痛点 4：不知道哪个 API 性价比高
- **现状**：没有历史数据对比，无法判断哪个 API 更划算
- **后果**：可能一直在用贵的 API，而便宜的更好
- **例子**：Agent 同时用两个天气 API，一个 0.01 美元/次，一个 0.005 美元/次，但你不知道
- **本项目方案**：记录每次购买的 token 类型、价格、用途，支持性价比分析

### 2.2 本项目方案总结

**x402 提供自动支付能力，Pact 提供预算控制，链上合约提供支出记录，三者结合实现"可控的 Agent 采购"。**

## 三、系统架构

### 3.1 整体架构图

```
┌─────────────────────────────────────────────────────────┐
│                  Agent Token 采购系统                     │
│                                                          │
│  ┌──────────┐                    ┌──────────────┐        │
│  │  你的    │   需要 token       │  Token       │        │
│  │  Agent   │ ─────────────────→ │  服务商      │        │
│  │          │                    │  (x402 API)  │        │
│  └────┬─────┘                    └──────┬───────┘        │
│       │                                 │                 │
│       │ ① GET /api/token                │                 │
│       │ ② 402 {price, token_type}       │                 │
│       │ ③ 检查预算 (Pact)               │                 │
│       │ ④ 支付 (CAW Wallet)            │                 │
│       │ ⑤ 收到 token                    │                 │
│       │ ⑥ 记录交易 (链上合约)           │                 │
│       ▼                                 ▼                 │
│  ┌──────────┐                    ┌──────────────┐        │
│  │  Pact    │                    │  支付记录    │        │
│  │  权限    │                    │  合约        │        │
│  │  控制    │                    │              │        │
│  └──────────┘                    └──────────────┘        │
└─────────────────────────────────────────────────────────┘
```

### 3.2 链路详解

| 步骤 | 动作 | 协议 | 说明 |
|------|------|------|------|
| ① | Agent 发 GET 请求 | HTTP | 请求某个付费 API（如 OpenAI、金融数据） |
| ② | 服务端返回 402 | x402 | 包含价格、token 类型、收款地址 |
| ③ | Agent 检查预算 | Pact | 查询本月已用预算，判断是否超限 |
| ④ | Agent 自动支付 | x402 + CAW | 预算内 → 自动支付；超预算 → 拒绝 |
| ⑤ | 服务端返回 token | HTTP | Agent 收到 API 调用次数或数据 |
| ⑥ | Agent 记录交易 | 链上合约 | 写入：时间、token 类型、金额、服务商 |

### 3.3 关键设计决策

#### 决策 1：预算检查在支付前还是支付后？
- **选择**：支付前
- **原因**：避免"付了钱才发现超预算"
- **权衡**：多一次链上查询，但能有效控制成本

#### 决策 2：交易记录存链上还是链下？
- **选择**：链上
- **原因**：不可篡改，任何人可查，便于审计
- **权衡**：gas 成本更高，但信任度最高

#### 决策 3：预算控制粒度？
- **选择**：单笔 + 每日 + 每月
- **原因**：多层防护，避免单笔大额 + 累计超支
- **权衡**：配置更复杂，但更安全

## 四、核心功能设计

### 4.1 自动购买 Token

**场景**：Agent 需要调用某个付费 API（比如获取金融数据、AI 模型 token 等）

**流程**：
1. Agent 请求 `GET /api/financial-data`
2. 服务端返回 `402`：需要支付 0.01 ETH
3. Agent 检查 Pact 预算 → 允许
4. Agent 通过 CAW 钱包支付
5. Agent 收到数据/token
6. 交易记录写入链上合约

**你能用的场景**：
- Agent 自动购买 OpenAI API 调用次数
- Agent 购买专业数据 token（金融、天气、新闻等）
- Agent 购买计算资源 token

### 4.2 支出记录与查询

**解决什么问题**：知道 Agent 花了多少钱、花在哪

**链上记录的数据**：
- **timestamp**：购买时间
- **token_type**：购买的 token 类型（如 "openai-api", "financial-data"）
- **amount**：支付金额
- **seller**：服务商地址
- **tx_hash**：交易哈希
- **purpose**：用途描述（可选）

**查询功能**：
- 按时间范围查询：本月花了多少？
- 按 token 类型查询：OpenAI API 花了多少？
- 按服务商查询：哪个服务商收了最多钱？
- 总支出统计：本月总支出、本季度总支出

### 4.3 预算权限控制（Pact）

**解决什么问题**：防止 Agent 乱花钱

**权限配置**：
- **per_tx_limit**：单笔交易上限（如 0.05 ETH）
- **daily_limit**：每日交易上限（如 0.2 ETH）
- **monthly_limit**：每月交易上限（如 2 ETH）
- **allowed_token_types**：允许购买的 token 类型（如 "openai-api", "financial-data"）
- **banned_token_types**：禁止购买的 token 类型（如 "gambling"）
- **warn_at_percent**：预算使用百分比警告（如 80%）
- **pause_at_percent**：预算使用百分比暂停（如 100%）

**你能控制的**：
- 设置每月预算上限
- 设置单笔交易上限
- 限制 Agent 只能购买特定类型的 token
- 预算快用完时收到警告
- 预算用完自动暂停购买

| 组件 | 技术 | 用途 |

## 七、你能用的场景举例

### 场景 1：Agent 自动续费 OpenAI API

```
你的 Agent 每天需要调用 OpenAI API 1000 次
→ 服务商返回 402：需要支付 0.01 ETH
→ Agent 检查本月预算（已用 0.5 ETH，上限 2 ETH）→ 允许
→ Agent 自动支付
→ 收到 API 调用次数
→ 链上记录：2026-05-30, openai-api, 0.01 ETH
```

### 场景 2：预算用完自动暂停

```
你的 Agent 本月已花费 1.98 ETH（上限 2 ETH）
→ 服务商返回 402：需要支付 0.05 ETH
→ Agent 检查预算 → 超出上限
→ Agent 拒绝支付，记录日志
→ 你收到通知：本月预算已用完
```

### 场景 3：查询支出报告

```
你问：本月 Agent 花了多少钱？
→ 查询链上记录
→ 返回：
  - OpenAI API: 0.8 ETH (80 次)
  - 金融数据: 0.5 ETH (50 次)
  - 天气数据: 0.2 ETH (20 次)
  - 总计: 1.5 ETH
```

### 场景 4：性价比分析

```
你问：哪个天气 API 更划算？
→ 查询链上记录
→ 返回：
  - 天气 API A: 0.01 ETH/次，共 100 次，总花费 1 ETH
  - 天气 API B: 0.005 ETH/次，共 100 次，总花费 0.5 ETH
  - 建议：切换到 API B，每月可节省 0.5 ETH
```

## 九、风险与缓解

| 风险 | 缓解 |
|------|------|
| CAW SDK 对合约调用的支持程度 | 先验证 CAW 能调自定义合约；不行就用 viem 直接签 |
| 测试网 ETH 不够 | Sepolia faucet + 多账号分批领 |
| 合约 gas 优化 | 测试网不要求——先跑通，再优化 |
| Agent 逻辑错误导致重复购买 | 添加幂等性检查，同一请求只支付一次 |
| 预算配置错误导致无法购买 | 提供默认配置，支持动态调整 |
